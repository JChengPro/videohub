## 总纲背诵版
> 这个项目是一个基于 Go 的短视频 Feed 流后端系统，整体采用 router -> handler ->  
service -> repository 分层结构。  
handler 负责接收请求和返回响应，service 负责业务逻辑，repository 负责操作  
MySQL。  
MySQL 存核心业务数据，Redis 缓存热点 Feed 数据，RabbitMQ + Worker 处理异步任务，  
比如视频发布后的缓存失效、点赞/取消点赞的异步写库。
>

###### http还是https的判断
```plain
func buildAbsoluteURL(c *gin.Context, p string) string {
    scheme := "http"                 // 默认按 http
    if c.Request.TLS != nil {        // 当前连接是 TLS，就认为是 https
        scheme = "https"
    }
    //用户 --HTTPS--> Nginx --HTTP--> Go，使用了反向代理，没有TLS但是仍然是HTTPS，
    if xf := c.GetHeader("X-Forwarded-Proto"); xf != "" { // 如果代理告诉了原始协议，就以它为准
        scheme = xf
    }
    return fmt.Sprintf("%s://%s%s", scheme, c.Request.Host, p)
}
```

一句话总结：

+ `c.Request.TLS != nil`：判断**当前到 Go 服务这一跳**是不是 HTTPS 
+ `X-Forwarded-Proto`：判断**用户最初请求时**是不是 HTTPS，主要给反向代理场景用

###### http和https的联系和区别
 HTTP 是传输网页数据的协议，HTTPS 则是在 HTTP 的基础上加了 TLS/SSL 加密，所以两者的联系是 **HTTPS = HTTP + 加密安全层**，区别是 **HTTP 明文传输不安全，HTTPS 加密传输更安全并能验证身份**。

###### `Authorization` 里前半段写 `Bearer`，是因为它表示后面带的是一种“持有即可使用”的 token，这是一种标准的 HTTP 认证方案，不是随便约定的字符串。  
###### 密码存储方法
本项目使用 bcrypt 对用户密码做不可逆哈希存储。注册时通过 GenerateFromPassword 生成密码哈希并保存到 MySQL；登录时通过 CompareHashAndPassword 校验用户输入的密码是否和数据库中的哈希匹配。bcrypt 自带盐值和计算成本，能降低彩虹表和暴力破解风险。

JWT的三部分组成

> JWT 由 Header、Payload、Signature 三部分组成。Header 描述 token 类型和签名算法；Payload 存放用户 ID、用户名、过期时间等声明信息；Signature 是用服务端密钥对 Header 和 Payload 计算出来的签名，用来防止 token 被篡改。Header 和 Payload 只是编码不  
是加密，所以不能放敏感信息。
>

###### 注册流程
  POST /account/register  
    -> Handler 解析 username/password  
    -> Service 校验参数  
    -> bcrypt 生成密码哈希  
    -> Repository 用 GORM 写入 accounts 表  
    -> 返回 account created

  核心点：数据库存的是密码哈希，不是明文密码。

######   登录流程
  POST /account/login  
    -> Handler 解析 username/password  
    -> Service 根据 username 查 MySQL  
    -> bcrypt 校验密码  
    -> 生成 JWT  
    -> 把 token 保存到 accounts.token  
    -> 返回 token 给前端

  核心点：JWT 返回给前端，后续请求通过 Authorization: Bearer token 携带。

  受保护接口鉴权流程

  请求带 Authorization: Bearer token  
    -> JWT 中间件解析 token  
    -> 得到 accountID/username  
    -> 查询 MySQL 对比当前 token  
    -> token 一致则 c.Set 当前用户信息  
    -> Handler 继续执行

  核心点：不是纯 JWT，会和数据库里的当前 token 对比，所以可以让旧 token 失效。

###### GORM 会从你传给 `Find` 的目标变量里，推断它对应的模型类型。你这里是：
```plain
var videos []video.Video

err := r.db.WithContext(ctx).
    Order("create_time desc").
    Limit(limit).
    Find(&videos).Error
```

`videos` 的元素类型是 `video.Video`，所以 GORM 会把它当成 `Video` 这个模型来查。默认规则下，GORM 会把结构体名转成 `snake_case` 再取复数作为表名，所以 `Video` 默认对应 `videos` 表。

也就是说，你这段代码大致等价于：

```plain
SELECT * FROM videos ORDER BY create_time DESC LIMIT ?
```

这里并不是因为变量名叫 `videos` 才查 `videos` 表，而是因为 **切片元素类型是 **`**video.Video**`，GORM 按模型类型映射到表名。

再强调一下，真正起作用的是这个类型：

```plain
[]video.Video
```

不是这个变量名：

```plain
videos
```

哪怕你改成：

```plain
var x []video.Video
db.Find(&x)
```

默认还是查 `videos` 表，因为模型还是 `Video`。表名规则看的是结构体类型，不看 Go 变量名。

当然，这个默认规则可以被改掉。

> Feed 最新列表第一页访问频率高，所以我给 /feed/latest 加了 Redis 缓存。请求进来后，如果是第一页，就先根据

GORM单条查询和数组查询的区别

GORM 常见搭配：

  First -> 查一条  
  Find  -> 查多条

  JSON 结果：

  *Video  -> {}  
  []Video -> []

  面试里可以说：

> 单条查询一般用于详情接口，使用 First，返回 *Video，查不到会有 record not found；列表查询使用 Find，返回  
[]Video，没有数据时通常返回空数组。前端根据返回的是对象还是数组来处理。

###### uniqueIndex:idx_video_account;not null
其中：

+ `uniqueIndex` 表示“唯一索引” 
+ `idx_like_video_account` 是这个索引的名字

###### GORM中Model()的作用

**告诉 GORM：“后面的这串操作，是基于哪个模型（表）来做的。”**

```plain
db.Model(&User{}).Where("active = ?", true).Update("name", "hello")
```

```plain
UPDATE users SET name='hello' WHERE active=true;
```

###### rabbitMQ如何实现视频发布后视频推荐流的更新
 前端请求最新视频流

  -> 后端先查 Redis

  -> Redis 有缓存，直接返回

  -> Redis 没缓存，再查 MySQL

  -> 查完 MySQL，把结果写入 Redis，缓存 30 秒

> feed:latest:* 是最新视频流的 Redis 缓存 key 前缀。因为 /feed/latest 查询结果会缓  
存到 Redis，发布新视频后旧缓存可能导致用户看不到最新内容，所以需要删除这类缓存。  
项目通过 RabbitMQ 发送 video_published 事件，让 worker 异步清理 feed:latest:* 缓  
存，实现 Feed 缓存的最终一致性。
>

  feed      ：表示这个 key 属于视频流模块  feed 更像是接口分类

  latest    ：表示这是“最新视频流”  也就是按发布时间倒序排列的视频列表

  limit=2   ：这次查询最多返回 2 条视频

  before=0  ：查询某个时间点之前的视频，0 表示第一页/最新一页

> feed:latest:limit=2:before=0 是最新视频流接口的 Redis 缓存 key。feed 表示视频流  
模块，latest 表示最新流，limit 表示每页数量，before 是基于时间的分页游标，  
before=0 表示第一页。因为不同分页参数返回结果不同，所以缓存 key 中必须包含 limit  
和 before。
>

###### 基础班worker更新视频缓存流
> 发布视频后，最新流缓存会变旧。worker 消费 video_published 事件后，通过 SCAN 查找  
所有 feed:latest:* 缓存 key，并批量删除。这样下一次访问 /feed/latest 时会重新查  
MySQL 并生成新缓存，保证 Feed 流最终一致。失败时通过 Nack 重新入队，避免临时  
Redis 故障导致缓存长期不一致。
>

###### d.Body和d.nack(false,true)的含义和作用
> d.Body 是 RabbitMQ 消息体，类型是 []byte，通常保存 JSON 数据，worker 会通过  
json.Unmarshal 把它解析成业务事件。d.Nack(false, false) 表示当前消息处理失败，并  
且不重新入队。第一个参数 multiple=false 表示只处理当前这一条消息，第二个参数  
requeue=false 表示不把消息放回队列，通常用于消息格式错误这类重试也无法恢复的问  
题。
>

RabbitMQ从外到内

> RabbitMQ 中消息先存放在 Queue 里，Go 调用 Consume 后会得到一个 deliveries 通道，  
worker 通过 for d := range deliveries 持续取出单条消息 d，其中 d.Body 是真正的消  
息内容，通常再反序列化成业务事件对象。
>

video_published 事件到 RabbitMQ。worker 进程消费这个事件后，异步删除

>

> 我对 RabbitMQ 的理解是，它是一个消息队列，主要由生产者、队列、消费者组成。生产者  
负责发送消息，队列负责暂存消息，消费者负责异步处理消息。项目中 API 服务是生产  
者，worker 服务是消费者。处理成功后会 Ack，失败时根据错误类型决定是否 Nack 重新  
入队。
>

> 简单来说，我在项目中用 RabbitMQ 实现了异步处理、削峰、解耦和最终一致性。核心数据  
比如视频发布仍然同步写 MySQL，非核心逻辑比如缓存失效、点赞写入等可以交给 worker

###### bcrypt 和加密有什么区别？
  
  bcrypt 是哈希，不是可逆加密。  
  加密一般是可逆的，拿到密钥后可以解密还原明文；bcrypt 是不可逆的，后端也无法从哈希  
  值反推出原密码。  
  在项目里，用户注册时我用 bcrypt.GenerateFromPassword 生成密码哈希，登录时用  
  CompareHashAndPassword 校验输入密码和数据库中的哈希是否匹配。密码存储场景应该用哈  
  希，不应该用可逆加密。

###### 为什么 Header/Payload 不能放敏感信息？

  别人拿到 token 后可以很容易解码看到里面内容，所以不能把密码、身份证号、手机号这类

  敏感信息放进去。

  项目里我只放了 account_id、username、过期时间这些必要信息。

###### localStorage 存 token 有什么风险？
  主要风险是 XSS。

  如果页面存在 XSS 漏洞，恶意脚本就可能读取 localStorage 里的 token，然后冒充用户发

  请求。

  所以把 token 放在 localStorage 使用方便，但安全性一般。更安全的做法通常是放在

  HttpOnly Cookie，这样前端脚本不能直接读取。

###### 限流设计
> **我在项目里增加了基于 Redis 的限流中间件（因为 Redis 读写快、支持过期时间、非常适合做短期高频计数。相比 MySQL，Redis 更轻量；相比本地内存，Redis 更适合后续多实例共享状态。）。核心思路是请求进来后，先根据 IP 或 accountID生成限流 key，再通过 Redis 做计数和过期控制，如果在时间窗口内超过阈值，就直接返回429。登录和注册接口按 IP 限流，点赞、评论、关注这类写接口按账号限流，这样可以防止暴力  
****请求和恶意刷接口。**
>
> **同一个 IP 短时间内不能疯狂登录**
>
>     - **同一个 IP 短时间内不能疯狂注册**
>     - **同一个账号短时间内不能疯狂点赞**
>     - **同一个账号短时间内不能疯狂评论**
>     - **同一个账号短时间内不能疯狂关注/取关**
>

今天真正学到的知识点

  这些是你今天最该带走的，而不是“又写了几个文件”。

+ yaml -> Go 配置结构体
    - os.ReadFile 读取文件
    - yaml.Unmarshal 解析成结构体
    - main.go 统一读配置，再把子配置传给各模块
+ 配置分层思想
    - main.go 负责装配
    - db.New(cfg.Database) 只负责建连接，不自己读配置文件
+ config.yaml 和 config.docker.yaml 的区别
    - 本地 go run 用本地配置
    - 程序跑在容器里时用 Docker 配置
+ pprof 是什么
    - Go 自带的性能分析工具
    - 用来看 CPU、内存、goroutine、阻塞等
    - 适合作为独立调试服务暴露，而不是业务接口
+ 限流的本质
    - Redis 做“带过期时间的请求计数器”
    - window 是时间窗口
    - maxRequests 是阈值
    - 登录/注册按 IP 限流，点赞/评论/关注按 accountID 限流
+ KeyFunc 的本质
    - 是函数类型，不是函数体
    - 通过传入不同函数，让同一个限流中间件支持按不同主体限流
+ JWTAuth 和 SoftJWTAuth 的区别
    - JWTAuth：没 token 或 token 非法，直接拦截
    - SoftJWTAuth：尽量解析，失败也放行
+ Bearer token 和 JWT claims 的区别
    - parseBearerToken：从 Header 里拆 token 字符串
    - auth.ParseToken：校验并解析 JWT 内容
+ token Redis 缓存的设计
    - MySQL 是权威状态
    - Redis 是鉴权缓存
    - 中间件优先读 Redis，miss 再回源 MySQL
+ Cache Aside（缓存旁路模式）
    - 先查缓存
    - miss 再查数据库
    - 查完回填缓存
+ 缓存失效原则
    - 哪个缓存依赖了被修改的数据，就删哪个缓存
    - 点赞会影响 video:detail:id=*
    - 发布/删除视频会影响 feed:latest:*

###### Dockerfile 和 docker-compose.study.yml 是什么关系
  一句话：

+ Dockerfile 解决：一个服务的镜像怎么做出来
+ docker-compose.study.yml 解决：多个服务怎么一起跑起来

先类比

  你可以把它理解成：

Dockerfile：做菜谱

docker-compose.yml：一桌菜怎么一起上桌你这个项目里

  你有这些服务：

+ mysql
+ redis
+ rabbitmq
+ api
+ worker

  其中：

+ mysql / redis / rabbitmq 已经有官方现成镜像，所以不用你写 Dockerfile
+ api / worker 是你自己的 Go 程序，所以要写 Dockerfile 告诉 Docker：
    - 代码在哪
    - 怎么编译
    - 入口程序是谁

  而 docker-compose.study.yml 再把它们组织起来：

+ 谁先启动
+ 端口怎么映射
+ 用哪个配置文件
+ 哪些目录要持久化
+ 哪些服务依赖哪些服务

  所以关系是：

> compose 里面会引用 Dockerfile 构建镜像，然后再把镜像变成容器运行。
>

  - 你的程序镜像构建逻辑写在 Dockerfile

  - 你的多服务编排逻辑写在 docker-compose.study.yml

######  一句话版
> 我用 Dockerfile 把 backend_study 编译成 API 和 worker 两个镜像，再用 docker  
compose 把 MySQL、Redis、RabbitMQ、API 和 worker 一起编排起来，方便本地一键启动和  
联调。
>

###### 结构版
> Dockerfile 主要分成两部分：第一部分用 Go 镜像编译 API 和 worker 二进制，第二部分用  
更小的 Alpine 作为运行环境，分别生成 API 和 worker 镜像。compose 负责把这些镜像和  
MySQL、Redis、RabbitMQ 组织起来，并处理端口、挂载卷、依赖顺序和健康检查。
>

######  为什么这样做
> 多阶段构建可以减小最终镜像体积；API 和 worker 共用一个 Dockerfile 可以减少重复；  
compose 则让整个项目依赖服务统一启动，避免手动逐个运行。
>

###### 你项目怎么用 Docker 的？
  我的项目里主要用 Docker 做本地环境统一和服务编排。  
  具体来说，我把 MySQL、Redis、RabbitMQ、API 和 worker 都放到 Docker Compose 里管理，  
  这样启动项目时不需要手动一个个安装和运行依赖，只需要一条 compose 命令就能把整套环境  
  拉起来。这样做的好处是环境更稳定，团队里不同机器上的运行方式也更一致。

  ———

###### Dockerfile 主要做了什么？
  我这个项目的 Dockerfile 主要做两件事：

1. 在构建阶段用 Go 官方镜像把后端代码编译成两个二进制：
    - API 程序
    - worker 程序
2. 在运行阶段用更小的 Alpine 镜像作为基础环境，把编译好的二进制拷进去，并准备配置目  
录和上传目录，最后分别指定 API 和 worker 的启动入口。

  所以它本质上是在描述：

> 我的 Go 项目怎么从源码变成可以直接运行的容器镜像。
>

  ———

###### 为什么要多阶段构建？
  因为 Go 的编译环境和最终运行环境没有必要放在同一个镜像里。

  如果直接用 Go 镜像运行程序，镜像会比较大，而且会带很多运行时根本用不到的编译工具。  
  所以我用了多阶段构建：

+ 第一阶段：负责编译
+ 第二阶段：只负责运行

  这样做的好处是：

+ 最终镜像更小
+ 更干净
+ 更适合部署
+ 安全性也更好一些

  ———

###### API 和 worker 为什么可以共用一个 Dockerfile？
  因为它们本质上属于同一个 Go 项目，只是入口程序不同。

  它们共享同一套：

+ go.mod
+ internal 业务代码
+ 配置目录
+ 构建环境

  区别只是：

+ API 编译的是 ./cmd
+ worker 编译的是 ./cmd/worker

  所以我在同一个 Dockerfile 里先把两个二进制都编译出来，然后分别定义：

+ target: api
+ target: worker

  这样可以减少重复，维护成本也更低。

  ———

###### Compose 是怎么把 MySQL、Redis、RabbitMQ、API、worker 串起来的？
  Compose 主要做的是服务编排。

  我在 compose 文件里定义了多个 service：

+ mysql
+ redis
+ rabbitmq
+ backend-study-api
+ backend-study-worker

  然后分别配置：

+ image 或 build
+ ports
+ volumes
+ environment
+ depends_on
+ healthcheck

  串起来的关键点有几个：

1. MySQL、Redis、RabbitMQ 用官方镜像启动
2. API 和 worker 用我自己的 Dockerfile 构建
3. API 和 worker 通过 depends_on + service_healthy 等待依赖服务健康后再启动
4. 配置文件通过 volume 挂载进容器
5. API 和 worker 在 Docker 网络里通过服务名访问依赖，比如：
    - mysql:3306
    - redis:6379
    - rabbitmq:5672

  所以 Compose 负责的是：

> 把单个容器组织成一套能协同工作的完整运行环境。
>

  ———

###### 本地配置和 Docker 配置有什么区别？
  区别主要在于：

> 程序运行的位置不同，依赖服务的访问地址也不同。
>

###### 本地配置 config.yaml
  用于我直接在宿主机上运行：

  go run ./cmd  
  go run ./cmd/worker

  这时候程序访问 MySQL、Redis、RabbitMQ，一般走宿主机映射端口，比如：

+ 127.0.0.1:3307
+ 127.0.0.1:6379
+ 127.0.0.1:5672

###### Docker 配置 config.docker.yaml
  用于 API 和 worker 自己也跑在容器里。

  这时候它们不能再访问 127.0.0.1，因为容器里的 127.0.0.1 指的是容器自己。  
  所以要改成通过 Docker Compose 的服务名访问：

+ mysql:3306
+ redis:6379
+ rabbitmq:5672

  所以两份配置本质上是为了适配：

+ 本地直接运行
+ 容器内运行

  这两个不同环境。



###### API程序和Worker程序的作用
 - API 程序：负责对外提供 HTTP 接口，处理注册登录、发布视频、查询 Feed、点赞评论等请

    求。

  - worker 程序：负责后台异步消费 RabbitMQ 消息，处理缓存失效、点赞写库、评论写库、关

###### 处理缓存失效是如何处理的
• 处理这类缓存失效的作用，是为了避免用户读到旧数据，并减少高频查询对 MySQL 的直接压

  力。项目里像 /feed/latest 这类接口会返回最新视频流，也就是用户进入首页后看到的视频流

  第一页；/video/detail 则是查询某个视频的详情页数据。这些接口都属于读多写少、访问频繁

  的场景，所以会优先走 Redis 缓存，这样可以更快返回结果、减轻数据库负载。但一旦底层数

  据发生变化，比如发布新视频、删除视频、点赞或取消点赞，原来的缓存结果就可能过期。如果

  不处理，用户可能会在 /feed/latest 里看不到刚发布的新视频，或者在 /video/detail 里看

  到旧的点赞数。所以项目采用的做法是：先让 MySQL 中的数据完成更新，再由 worker 异步删

  除受影响的 Redis key，例如发布或删除视频后删除 feed:latest:*，也就是删除“最新视频流

  第一页”等相关缓存；点赞或取消点赞后删除 video:detail:id=<videoID>，也就是删除某个视

  频的详情缓存。这样下一次用户再访问 /feed/latest 或 /video/detail 时，就会因为缓存未

  命中而重新查询 MySQL，并把最新结果写回 Redis，从而兼顾读取性能和数据一致性。
