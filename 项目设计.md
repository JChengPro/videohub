# VideoHub 项目设计

VideoHub 是一个视频内容社区，后端基于 Go + Gin + GORM，前端基于 Vue3 + Vite。项目目标是实现从账号、视频、点赞评论、关注关系到视频流浏览和容器化部署的一套完整后端工程。

## 1. 技术栈

| 维度 | 组件/工具 | 说明 |
| --- | --- | --- |
| 后端语言 | Go | API 与 Worker 双进程 |
| Web 框架 | Gin | 路由、参数绑定、中间件 |
| ORM | GORM | CRUD 与 AutoMigrate |
| 数据库 | MySQL 8 | 核心业务数据与 Outbox 消息 |
| 缓存 | Redis | token、限流、视频时间线、视频实体缓存、热榜 |
| 本地缓存 | go-cache | 视频实体 L1 本地内存缓存 |
| 消息队列 | RabbitMQ | 后置事件异步消费 |
| 前端 | Vue3 + Vite + TypeScript | 页面与接口联调 |
| 部署 | Docker Compose | 一键启动 MySQL、Redis、RabbitMQ、API、Worker、Frontend |

## 2. 总体架构

![整体架构](picture/整体架构.png)

```text
frontend
  -> Nginx /api 反向代理
backend API
  -> 参数校验 / JWT / 限流
  -> 同步写 MySQL
  -> 同事务写 outbox_msgs
worker
  -> Outbox Poller 发布 MQ
  -> RabbitMQ Consumer 处理缓存、热度、视频时间线
redis
  -> token 缓存、限流计数、视频时间线、视频实体缓存、热榜
mysql
  -> 业务数据最终存储
```

## 3. API + Worker 双进程

| 进程 | 入口 | 职责 |
| --- | --- | --- |
| API | `backend/cmd/main.go` | 提供 HTTP 接口，处理同步主链路 |
| Worker | `backend/cmd/worker/main.go` | 启动 Outbox Poller，消费 RabbitMQ 队列 |

API 和 Worker 共用同一个 `backend/Dockerfile`，通过多阶段构建分别输出 `/app/api` 和 `/app/worker`。Docker Compose 中 `backend` 服务使用 `target: api`，`worker` 服务使用 `target: worker`。

## 5. Outbox Pattern

### 作用

Outbox 用来解决“业务写库成功，但 MQ 发送失败”的双写一致性问题。

项目中点赞、评论、视频发布/删除都采用：

```text
同步写业务表
-> 同一个 MySQL 事务写 outbox_msgs
-> Poller 异步读取 outbox_msgs 并发布 RabbitMQ
-> Consumer 执行后置任务
```

### 状态流转

```text
pending
  -> publishing
  -> published
```

发送失败时：

```text
publishing -> pending
retry_count + 1
last_error 记录错误
```

如果 worker 崩溃导致消息长期停在 `publishing`，`ResetStalePublishingOutbox` 会把超时消息恢复为 `pending`。

### 并发处理

Poller 抢占消息时使用条件更新：

```text
UPDATE outbox_msgs
SET status = 'publishing'
WHERE id = ? AND status = 'pending'
```

MySQL 执行这类 `UPDATE` 时会对命中的行加排他锁，只有 `RowsAffected == 1` 的 worker 才算抢到消息。这个机制解决的是“多个 worker 同时扫描到同一条 outbox 记录”的问题。

Consumer 的幂等处理解决的是“同一条 MQ 消息可能被重复消费”的问题。两者关注点不同。

### 已接入事件

| 事件 | 事务内同步操作 | 异步后置操作 |
| --- | --- | --- |
| `video_published` | 写 `videos` + 写 outbox | 写 Redis 视频时间线，删除旧视频流缓存 |
| `video_deleted` | 删除 `videos` + 写 outbox | 从 Redis 视频时间线移除，删除旧视频流缓存 |
| `like_created` | 写 `likes`，更新 `videos.likes_count`，写 outbox | `popularity +1`，更新热榜，删详情缓存 |
| `like_deleted` | 删除 `likes`，更新 `videos.likes_count`，写 outbox | `popularity -1`，更新热榜，删详情缓存 |
| `comment_published` | 写 `comments`，写 outbox | `popularity +2`，更新热榜 |
| `comment_deleted` | 删除 `comments`，写 outbox | `popularity -2`，更新热榜 |

## 6. 最新视频流

### 6.1 Redis 时间线

最新流的热数据存在 Redis ZSET：

```text
key = Redis 视频时间线
member = video_id
score = 视频发布时间毫秒时间戳
```

视频发布后，worker 消费 `video_published`，把 `video_id` 和发布时间写入 Redis 视频时间线。视频删除后，worker 从 Redis 视频时间线移除对应 `video_id`。

时间线只保留最新 1000 条：

```text
按 rank 删除旧数据，只保留最新 1000 条
```

这样 Redis 只保存高频访问的最新视频，旧数据继续保存在 MySQL。

### 6.2 冷热分离

冷热分离的边界是 Redis 视频时间线中最老一条数据的 score：

```text
watermark = ZSET 最老视频的 create_time
```

读取最新视频流：

```text
如果 Redis 时间线为空：
  singleflight 只让一个请求从 MySQL 重建最近 1000 条

如果 reqTime <= watermark：
  已经翻到冷数据区，直接查 MySQL

如果 reqTime > watermark：
  还在热数据区，从 Redis ZSET 倒序取 video_id

如果 Redis 热数据不够一页：
  再从 MySQL 补齐剩余冷数据
```

### 6.3 singleflight

`singleflight` 用于进程内并发去重，避免缓存失效时重复查 MySQL。

| Key | 场景 |
| --- | --- |
| `sf:fallback:global_timeline_rebuild` | Redis 时间线为空时重建 |
| `sf:cold:listLatest:{limit}:{reqTime}` | 冷数据分页查库 |
| `sf:stitch:listLatest:{remainLimit}:{cursor}` | 冷热边界补数据 |
| `sf:entity:{videoID}` | 视频实体缓存 miss 后查 MySQL |

`singleflight` 内部由库 `golang.org/x/sync/singleflight` 实现，项目在视频流业务中定义 `requestGroup singleflight.Group` 并调用 `Do(key, fn)`。

### 6.4 三级缓存

Redis 时间线只保存视频 ID，完整视频信息通过 `GetVideoByIDs` 查询：

```text
L1：go-cache 本地内存缓存
L2：Redis，key = video:entity:{id}
L3：MySQL videos 表
```

L1 初始化：

```go
localcache.New(3*time.Second, 5*time.Second)
```

第一个参数是默认 TTL，第二个参数是过期数据清理间隔。项目写入视频实体时显式指定 `5*time.Second`，所以该场景下 L1 视频实体实际缓存约 5 秒。

L2 Redis 写入 TTL 为 1 小时。L3 MySQL 查询处使用 `singleflight`，多个 goroutine 写同一个 `videoMap` 时使用 `sync.Mutex` 保护，因为 Go map 并发写不安全。

最终通过 `buildOrderedResult` 按 Redis ZSET 返回的 video_id 顺序重新组装，避免 map 无序导致视频流顺序错乱。

## 7. 热榜

热榜使用 Redis ZSET：

```text
key = Redis 热榜 ZSET
member = video_id
score = popularity
```

点赞/取消点赞：

```text
popularity +1 / -1
Redis 热榜 ZSET 分数 +1 / -1
删除 video:detail 缓存
```

评论发布/删除：

```text
popularity +2 / -2
Redis 热榜 ZSET 分数 +2 / -2
```

热视频榜优先从 Redis ZSET 取 video_id，再回表查询视频详情并按 ZSET 顺序重排。Redis 没有数据时回退 MySQL。

## 8. 账号、鉴权和限流

![登录_鉴权_撤销token](picture/登录_鉴权_撤销token.png)

### JWT 鉴权

- 登录成功后生成 JWT。
- token 同时写入 MySQL 和 Redis。
- JWTAuth 强鉴权用于必须登录的接口。
- SoftJWTAuth 用于视频流等公开接口：不带 token 也能访问，带合法 token 时写入 `accountID`，用于返回 `is_liked`。
- 退出登录会清空 MySQL token 并删除 Redis token，使旧 token 立即失效。

### 接口限流

限流计数存在 Redis：

| 场景 | key 前缀 | 策略 |
| --- | --- | --- |
| 登录 | `account_login` | 按 IP，10 次/分钟 |
| 注册 | `account_register` | 按 IP，5 次/小时 |
| 点赞 | `like_write` | 按账号，30 次/分钟 |
| 评论 | `comment_write` | 按账号，10 次/分钟 |
| 关注 | `social_write` | 按账号，20 次/分钟 |
